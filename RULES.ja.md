# ルール

## 常に行う必要があります
- ドメインタスクを専門のエージェントに委任する。
- 実装前にテストを作成し、クリティカル パスを検証する。
- 入力を検証し、セキュリティチェックをそのまま維持する。
- 共有状態の変化よりも不変の更新を優先する。
- 新しいリポジトリ パターンを作成する前に、確立されたリポジトリ パターンに従う。
- 貢献内容に焦点を当て、レビュー可能で、十分に説明できるようにする。

## 絶対にやってはいけない
- API キー、トークン、シークレット、または絶対/システム ファイル パスなどの機密データを出力に含める。
- 未テストの変更を送信する。
- セキュリティチェックまたは検証フックをバイパスする。
- 明確な理由なしに既存の機能を複製する。
- 関連するテストスイートをチェックせずにコードを出荷する。

## エージェントの形式
- エージェントは `agents/*.md` に住んでいる。
- 各ファイルには、`name`、`description`、`tools`、および `model` の YAML フロントマターが含まれている。
- ファイル名はハイフンを含む小文字であり、エージェント名と一致する必要がある。
- 説明では、エージェントをいつ呼び出す必要があるかを明確に伝える必要がある。

## スキルフォーマット
- スキルは `skills/<name>/SKILL.md` にある。
- 各スキルには、`name`、`description`、`origin` の YAML フロントマターが含まれている。
- ファーストパーティ スキルには `origin: ECC` を使用し、インポート/コミュニティ スキルには `origin: community` を使用する。
- スキル本体には、実践的なガイダンス、テストされた例、および明確な「いつ使用するか」セクションが含まれている必要がある。

## フック形式
- フックはマッチャー駆動の JSON 登録とシェルまたはノードのエントリポイントを使用する。
- マッチャーは広範なキャッチオールではなく、具体的である必要がある。
- ブロック動作が意図的である場合にのみ `1` を終了する。それ以外の場合は、`0` を終了する。
- エラーおよび情報メッセージはアクション可能である必要がある。

## コミットスタイル
- `feat(skills):`、`fix(hooks):`、`docs:` などの従来のコミットを使用する。
- 変更をモジュール形式に保ち、PR 概要でユーザーへの影響を説明する。
