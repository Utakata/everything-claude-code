# Rules

## Must Always
- ドメインタスクは特化型エージェントに委譲する。
- 実装前にテストを記述し、クリティカルパスを検証する。
- 入力を検証し、セキュリティチェックを維持する。
- 共有状態の変更よりも、イミュータブルな更新を優先する。
- 新しいパターンを発明する前に、確立されたリポジトリのパターンに従う。
- 貢献を焦点を絞った、レビュー可能な、説明の行き届いたものに保つ。

## Must Never
- APIキー、トークン、シークレット、絶対パス/システムファイルパスなどの機密データをの出力に含めない。
- テストされていない変更を提出しない。
- セキュリティチェックやバリデーションフックをバイパスしない。
- 明確な理由なく既存の機能を複製しない。
- 関連するテストスイートを確認せずにコードを出荷しない。

## Agent Format
- エージェントは `agents/*.md` に配置する。
- 各ファイルには、`name`, `description`, `tools`, `model` を含む YAMLフロントマターを含める。
- ファイル名はハイフン区切りの小文字とし、エージェント名と一致させること。
- 説明は、エージェントを呼び出すべき状況を明確に伝える必要がある。

## Skill Format
- スキルは `skills/<name>/SKILL.md` に配置する。
- 各スキルには、`name`, `description`, `origin` を含む YAMLフロントマターを含める。
- ファーストパーティスキルには `origin: ECC` を、インポート/コミュニティスキルには `origin: community` を使用する。
- スキル本文には、実践的なガイダンス、テスト済みの例、明確な「いつ使うべきか (When to Use)」セクションを含める。

## Hook Format
- フックは、マッチャー駆動のJSON登録と、シェルまたはNodeエントリポイントを使用する。
- マッチャーは、広範囲のキャッチオールではなく、具体的である必要がある。
- ブロッキング動作が意図的な場合のみ `1` で終了し、それ以外の場合は `0` で終了する。
- エラーメッセージや情報メッセージは、アクション可能なものであること。

## Commit Style
- `feat(skills):`, `fix(hooks):`, `docs:` などのコンベンショナルコミットを使用する。
- 変更はモジュール化し、PRのサマリーでユーザーへの影響を説明する。
