# Everything Claude Code (ECC) — エージェントの指示

これはソフトウェア開発向けの60の特化型エージェント、231のスキル、75のコマンド、および自動化されたhookワークフローを提供する**本番環境対応のAIコーディングプラグイン**である。

**Version:** 2.0.0-rc.1

## コア原則

1. **Agent-First** — ドメインタスクを特化型エージェントに委譲する
2. **Test-Driven** — 実装前にテストを書くこと。80%以上のカバレッジが必須である
3. **Security-First** — セキュリティを決して妥協しないこと。すべての入力を検証する
4. **Immutability** — 常に新しいオブジェクトを作成し、既存のものを決して変更（mutate）しない
5. **Plan Before Execute** — コードを書く前に複雑な機能の計画を立てる

## エージェントのオーケストレーション

ユーザーからのプロンプトなしに自発的にエージェントを使用すること:
- 複雑な機能の要求 → **planner**
- コードを記述/変更した直後 → **code-reviewer**
- バグ修正または新機能 → **tdd-guide**
- アーキテクチャの決定 → **architect**
- セキュリティに敏感なコード → **security-reviewer**
- 自律型ループ / ループ監視 → **loop-operator**
- ハーネス設定の信頼性とコスト → **harness-optimizer**

独立した操作には並列実行を使用し、複数のエージェントを同時に起動する。

## セキュリティガイドライン

**いかなるコミットの前にも:**
- シークレット（APIキー、パスワード、トークン）をハードコードしない
- すべてのユーザー入力を検証する
- SQLインジェクションの防止（パラメータ化クエリ）
- XSSの防止（サニタイズされたHTML）
- CSRF保護の有効化
- 認証/認可の検証
- すべてのエンドポイントでのレート制限
- エラーメッセージから機密データを漏洩させない

**シークレット管理:** シークレットを絶対にハードコードしないこと。環境変数またはシークレットマネージャーを使用する。起動時に必須シークレットを検証する。漏洩したシークレットは直ちにローテーションする。

**セキュリティ上の問題が発見された場合:** 停止 → security-reviewer エージェントを使用 → 致命的（CRITICAL）な問題を修正 → 漏洩したシークレットをローテーション → 同様の問題がないかコードベースを確認する。

## コーディングスタイル

**Immutability（重要）:** 常に新しいオブジェクトを作成し、決して変更（mutate）しない。変更を適用した新しいコピーを返す。

**ファイル構成:** 少数で大規模なファイルよりも、多数で小規模なファイルを優先する。通常は200〜400行、最大800行とする。種類ではなく、機能/ドメインごとに構成する。高凝集度、低結合度。

**エラーハンドリング:** すべてのレベルでエラーを処理する。UIコードではユーザーフレンドリーなメッセージを提供する。サーバーサイドでは詳細なコンテキストをログに記録する。エラーを暗黙のうちに握りつぶしてはならない。

**入力検証:** システムの境界ですべてのユーザー入力を検証する。スキーマベースの検証を使用する。明確なメッセージと共にフェイルファスト（fail-fast）させる。外部データを決して信用しない。

**コード品質チェックリスト:**
- 関数は小さく（50行未満）、ファイルは焦点を絞る（800行未満）
- 深いネストを避ける（4レベルまで）
- 適切なエラーハンドリングを行い、ハードコードされた値を避ける
- 読みやすく、適切な名前の識別子を使用する

## テスト要件

**最小カバレッジ: 80%**

テストタイプ（すべて必須）:
1. **ユニットテスト** — 個々の関数、ユーティリティ、コンポーネント
2. **統合テスト** — APIエンドポイント、データベース操作
3. **E2Eテスト** — 重要なユーザーフロー

**TDDワークフロー（必須）:**
1. 最初にテストを書く（RED） — テストは失敗しなければならない
2. 最小限の実装を書く（GREEN） — テストは成功しなければならない
3. リファクタリング（IMPROVE） — カバレッジが80%以上であることを確認する

失敗時のトラブルシューティング: テストの分離を確認 → モックを検証 → 実装を修正する（テストが間違っている場合を除き、テストは修正しない）。

## 開発ワークフロー

1. **Plan** — planner エージェントを使用し、依存関係とリスクを特定し、フェーズに分割する
2. **TDD** — tdd-guide エージェントを使用し、最初にテストを書き、実装し、リファクタリングする
3. **Review** — code-reviewer エージェントを直ちに使用し、CRITICAL/HIGHな問題に対処する
4. **適切な場所での知識の共有**
   - 個人のデバッグメモ、環境設定、および一時的なコンテキスト → 自動メモリ（auto memory）
   - チーム/プロジェクトの知識（アーキテクチャの決定、APIの変更、ランブック） → プロジェクトの既存のドキュメント構造
   - 現在のタスクによって関連するドキュメントやコードコメントが既に生成されている場合、同じ情報を他の場所で複製しない
   - プロジェクトのドキュメントに明確な配置場所がない場合は、新しいトップレベルファイルを作成する前に確認する
5. **Commit** — Conventional Commitsフォーマット、包括的なPRサマリー

## ワークフローサーフェスポリシー

- `skills/` は正規のワークフローサーフェスである。
- 新しいワークフローの貢献は、まず `skills/` に配置する必要がある。
- `commands/` はレガシーなスラッシュエントリの互換性サーフェスであり、移行やクロスハーネスのパリティのためにシム（shim）が引き続き必要な場合にのみ追加または更新する必要がある。

## Gitワークフロー

**コミットフォーマット:** `<type>: <description>` — Types: feat, fix, refactor, docs, test, chore, perf, ci

**PRワークフロー:** 完全なコミット履歴を分析する → 包括的なサマリーを起草する → テスト計画を含める → `-u` フラグを使用してプッシュする。

## アーキテクチャパターン

**APIレスポンスフォーマット:** 成功インジケーター、データペイロード、エラーメッセージ、およびページネーションのメタデータを含む一貫したエンベロープ。

**リポジトリパターン:** 標準的なインターフェース（findAll, findById, create, update, delete）の背後にデータアクセスをカプセル化する。ビジネスロジックは抽象インターフェースに依存し、ストレージメカニズムには依存しない。

**スケルトンプロジェクト:** 実績のあるテンプレートを検索し、並行エージェント（セキュリティ、拡張性、関連性）で評価し、最も適したものをクローンし、証明された構造内で反復（iterate）する。

## パフォーマンス

**コンテキスト管理:** 大規模なリファクタリングや複数機能のコンテキストウィンドウの最後の20%を避ける。機密性の低いタスク（単一の編集、ドキュメント、単純な修正）では高い使用率が許容される。

**ビルドのトラブルシューティング:** build-error-resolver エージェントを使用する → エラーを分析する → 段階的に修正する → 修正ごとに検証する。

## プロジェクト構造

```
agents/          — 60の特化型サブエージェント
skills/          — 231のワークフロースキルとドメイン知識
commands/        — 75のスラッシュコマンド
hooks/           — トリガーベースの自動化
rules/           — 常に従うべきガイドライン（共通 + 言語別）
scripts/         — クロスプラットフォームのNode.jsユーティリティ
mcp-configs/     — 14のMCPサーバー設定
tests/           — テストスイート
```

`commands/` は互換性のためにリポジトリに残されているが、長期的な方向性としては skills-first である。

## 成功指標

- すべてのテストが80%以上のカバレッジで合格する
- セキュリティの脆弱性がない
- コードが読みやすく、保守可能である
- パフォーマンスが許容範囲内である
- ユーザー要件を満たしている
