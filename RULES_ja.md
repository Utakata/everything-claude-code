# Rules

## Must Always
- ドメインのタスクは特化型エージェントに委譲すること。
- 実装前にテストを記述し、クリティカルパスを検証すること。
- 入力を検証し、セキュリティチェックを維持すること。
- 共有状態のミューテーションよりも不変（immutable）な更新を優先すること。
- 新たなパターンを発明する前に、リポジトリの既存のパターンに従うこと。
- コントリビューションは焦点を絞り、レビュー可能であり、適切に説明されている状態を保つこと。

## Must Never
- APIキー、トークン、シークレット、絶対パス/システムファイルパスなどの機密データを出力に含めないこと。
- テストされていない変更をサブミットしないこと。
- セキュリティチェックやバリデーションのフックをバイパスしないこと。
- 明確な理由なく既存の機能を複製しないこと。
- 関連するテストスイートの確認なしにコードを出荷しないこと。

## Agent Format
- エージェントは `agents/*.md` に配置する。
- 各ファイルには、`name`、`description`、`tools`、`model` を含む YAML フロントマターを含めること。
- ファイル名はハイフン区切りの小文字とし、エージェント名と一致させること。
- 説明（description）には、そのエージェントをいつ呼び出すべきかを明確に伝えること。

## Skill Format
- スキルは `skills/<name>/SKILL.md` に配置する。
- 各スキルには、`name`、`description`、`origin` を含む YAML フロントマターを含めること。
- ファーストパーティのスキルには `origin: ECC` を使用し、インポート/コミュニティのスキルには `origin: community` を使用すること。
- スキルの本文には、実践的なガイダンス、テスト済みの例、そして明確な「When to Use (いつ使うか)」のセクションを含めること。

## Hook Format
- フックはマッチャー駆動のJSON登録と、シェルまたはNodeのエントリポイントを使用する。
- マッチャーは、広範なキャッチオールではなく、具体的なものにすること。
- ブロッキングする動作が意図的である場合にのみ `1` で終了し、それ以外の場合は `0` で終了すること。
- エラーおよび情報メッセージは、アクション可能（actionable）であること。

## Commit Style
- `feat(skills):`、`fix(hooks):`、`docs:` などの Conventional Commits を使用すること。
- 変更はモジュール化し、ユーザーに影響を与える内容をPRのサマリーで説明すること。
