# Rules

## Must Always
- ドメインタスクのために特化型エージェントに委任すること。
- 実装の前にテストを書き、クリティカルパスを検証すること。
- 入力を検証し、セキュリティチェックを無傷のまま保つこと。
- 共有状態を変更するよりも、不変な更新を優先すること。
- 新しいものを発明する前に、確立されたリポジトリパターンに従うこと。
- コントリビューションは焦点を絞り、レビュー可能で、十分に説明された状態に保つこと。

## Must Never
- APIキー、トークン、シークレット、または絶対/システムファイルパスなどの機密データを出力に含めないこと。
- テストされていない変更を提出しないこと。
- セキュリティチェックや検証フックをバイパスしないこと。
- 明確な理由なしに既存の機能を複製しないこと。
- 関連するテストスイートを確認せずにコードを出荷しないこと。

## Agent Format
- エージェントは `agents/*.md` に置かれる。
- 各ファイルには、`name`、`description`、`tools`、および `model` を含むYAMLフロントマターが含まれる。
- ファイル名は小文字でハイフン区切りであり、エージェント名と一致しなければならない。
- 説明は、エージェントがいつ呼び出されるべきかを明確に伝えなければならない。

## Skill Format
- スキルは `skills/<name>/SKILL.md` に置かれる。
- 各スキルには、`name`、`description`、および `origin` を含むYAMLフロントマターが含まれる。
- ファーストパーティのスキルには `origin: ECC` を使用し、インポート/コミュニティスキルには `origin: community` を使用すること。
- スキル本体には、実践的なガイダンス、テストされた例、および明確な「いつ使用するか（When to Use）」セクションが含まれるべきである。

## Hook Format
- フックはマッチャー駆動のJSON登録とシェルまたはNodeのエントリポイントを使用する。
- マッチャーは広範なキャッチオールではなく、具体的であるべきである。
- 動作を意図的にブロックする場合にのみ `1` で終了すること。それ以外の場合は `0` で終了すること。
- エラーおよび情報メッセージは、アクション可能であるべきである。

## Commit Style
- `feat(skills):`、`fix(hooks):`、または `docs:` のようなConventional Commitsを使用すること。
- 変更はモジュール化された状態に保ち、PRサマリーでユーザー向けの影響を説明すること。
