# Rules

## Must Always
- ドメインタスクは専門のagentsに委任すること。
- 実装の前にテストを書き、重要なパス（critical paths）を検証すること。
- 入力を検証し、セキュリティチェックを無傷のまま保つこと。
- 共有状態を変更（mutate）するよりも、イミュータブル（不変）な更新を優先すること。
- 新しいパターンを発明する前に、確立されたリポジトリパターンに従うこと。
- 貢献（contributions）は焦点を絞り、レビュー可能で、適切に説明された状態を保つこと。

## Must Never
- 出力に、APIキー、トークン、シークレット、絶対/システムファイルパスなどの機密データを含めないこと。
- テストされていない変更を提出しないこと。
- セキュリティチェックや検証のためのhooksをバイパスしないこと。
- 明確な理由なしに既存の機能を複製しないこと。
- 関連するテストスイートを確認せずにコードを出荷（ship）しないこと。

## Agent Format
- Agentsは`agents/*.md`に配置される。
- 各ファイルには、`name`、`description`、`tools`、および`model`を含むYAMLフロントマターが含まれる。
- ファイル名はハイフンを使った小文字であり、agentの名前と一致しなければならない。
- 説明（description）には、そのagentがいつ呼び出されるべきかを明確に伝えなければならない。

## Skill Format
- Skillsは`skills/<name>/SKILL.md`に配置される。
- 各skillには、`name`、`description`、および`origin`を含むYAMLフロントマターが含まれる。
- ファーストパーティのskillsには`origin: ECC`を使用し、インポート/コミュニティのskillsには`origin: community`を使用すること。
- skillの本文には、実践的なガイダンス、テスト済みの例、および明確な「When to Use（いつ使用するか）」のセクションを含めるべきである。

## Hook Format
- Hooksは、マッチャー（matcher）駆動のJSON登録と、シェルまたはNodeのエントリーポイントを使用する。
- マッチャーは、広範なキャッチオール（catch-alls）ではなく、具体的であるべきである。
- 意図的に動作をブロックする場合にのみ`1`で終了し、それ以外の場合は`0`で終了すること。
- エラーおよび情報メッセージは、アクション可能（actionable）であるべきである。

## Commit Style
- `feat(skills):`、`fix(hooks):`、または`docs:`のようなconventional commitsを使用すること。
- 変更はモジュール化された状態を保ち、PRの要約でユーザーに与える影響を説明すること。
