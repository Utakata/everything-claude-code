# ルール (Rules)

## 常にすべきこと (Must Always)
- ドメインタスクは特化型エージェントに委譲すること。
- 実装前にテストを書き、クリティカルパスを検証すること。
- 入力を検証し、セキュリティチェックを無傷のまま維持すること。
- 共有状態の変更（mutate）よりも不変な（immutable）更新を優先すること。
- 新しいパターンを発明する前に、確立されたリポジトリパターンに従うこと。
- 貢献は焦点を絞り、レビュー可能で、適切に説明された状態に保つこと。

## 決してしてはならないこと (Must Never)
- APIキー、トークン、シークレット、または絶対/システムファイルパスなどの機密データを出力に含めないこと。
- テストされていない変更を提出しないこと。
- セキュリティチェックやバリデーションフックをバイパスしないこと。
- 明確な理由なしに既存の機能を重複させないこと。
- 関連するテストスイートを確認せずにコードを出荷しないこと。

## エージェントのフォーマット (Agent Format)
- エージェントは `agents/*.md` に配置する。
- 各ファイルには、`name`, `description`, `tools`, `model` を含む YAML フロントマターを含める。
- ファイル名はハイフン付きの小文字とし、エージェント名と一致させること。
- Description には、エージェントがいつ呼び出されるべきかを明確に伝えること。

## スキルのフォーマット (Skill Format)
- スキルは `skills/<name>/SKILL.md` に配置する。
- 各スキルには、`name`, `description`, `origin` を含む YAML フロントマターを含める。
- ファーストパーティスキルには `origin: ECC` を使用し、インポート/コミュニティスキルには `origin: community` を使用する。
- スキルの本文には、実践的なガイダンス、テスト済みの例、および明確な「いつ使用するか (When to Use)」のセクションを含めること。

## フックのフォーマット (Hook Format)
- フックはマッチャー駆動の JSON 登録と、シェルまたは Node のエントリーポイントを使用する。
- マッチャーは、広範なキャッチオールではなく、具体的なものにすること。
- ブロック動作が意図的である場合にのみ `exit 1` とし、それ以外の場合は `exit 0` とすること。
- エラーおよび情報メッセージは、実行可能な（actionable）ものにすること。

## コミットスタイル (Commit Style)
- `feat(skills):`, `fix(hooks):`, `docs:` などの Conventional Commits を使用すること。
- 変更をモジュール化し、PRのサマリーでユーザーへの影響を説明すること。
