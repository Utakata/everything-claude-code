# Everything Claude Code (ECC) — エージェントの指示

これは、**本番環境に対応した AI コーディング プラグイン**であり、ソフトウェア開発用に 47 の専門エージェント、181 のスキル、79 のコマンド、自動化されたフック ワークフローを提供する。

**バージョン:** 1.10.0

## 基本原則

1. **エージェントファースト** — ドメインタスクを専門のエージェントに委任します
2. **テスト駆動** — 実装前にテストを作成、80% 以上のカバレッジが必要
3. **セキュリティ第一** — セキュリティについては決して妥協しない。すべての入力を検証する
4. **不変性** — 常に新しいオブジェクトを作成し、既存のオブジェクトを変更しない
5. **実行前に計画を立てる** — コードを記述する前に複雑な機能を計画する

## エージェント オーケストレーション

ユーザー プロンプトを表示せずにエージェントを積極的に使用する。
- 複雑な機能リクエスト → **プランナー**
- 作成/変更されたばかりのコード → **コードレビューア**
- バグ修正または新機能 → **tdd-guide**
- アーキテクチャの決定 → **建築家**
- セキュリティに敏感なコード → **security-reviewer**
- 自律ループ / ループ監視 → **ループオペレーター**
- ハーネス構成の信頼性とコスト → **harness-optimizer**

独立した操作には並列実行を使用する。複数のエージェントを同時に起動する。

## セキュリティガイドライン

**コミット前:**
- ハードコードされたシークレット (API キー、パスワード、トークン) はありません
- すべてのユーザー入力が検証されました
- SQL インジェクションの防止 (パラメータ化されたクエリ)
- XSS防止（HTMLのサニタイズ）
- CSRF保護が有効になっています
- 認証/認可が検証済み
- すべてのエンドポイントでのレート制限
- エラー メッセージによって機密データが漏洩しない

**シークレット管理:** シークレットを決してハードコーディングしないでください。環境変数またはシークレット マネージャーを使用する。起動時に必要なシークレットを検証する。公開された秘密は直ちにローテーションする。

**セキュリティ問題が見つかった場合:** 停止→セキュリティレビューアーエージェントを使用→重大な問題を修正→公開されたシークレットをローテーション→同様の問題がないかコードベースをレビューする。

## コーディングスタイル

**不変性 (クリティカル):** 常に新しいオブジェクトを作成し、決して変更しないでください。変更を適用した新しいコピーを返する。

**ファイル構成:** 大きなファイルはほとんどなく、小さなファイルが多数ある。通常 200 ～ 400 行、最大 800 行タイプごとではなく、機能/ドメインごとに整理する。凝集性が高く、結合性が低い。

**エラー処理:** あらゆるレベルでエラーを処理する。 UI コードでユーザーフレンドリーなメッセージを提供する。詳細なコンテキストをサーバー側でログに記録する。決してエラーを黙って飲み込んではいけない。

**入力検証:** システム境界ですべてのユーザー入力を検証する。スキーマベースの検証を使用する。明確なメッセージで迅速に失敗する。外部データを決して信用しないでください。

**コード品質チェックリスト:**
- 小規模な機能 (<50 lines), files focused (<800 lines)
- No deep nesting (>4 レベル)
- 適切なエラー処理、ハードコーディングされた値なし
- 読みやすく、適切な名前の識別子

## テスト要件

**最小カバレッジ: 80%**

テストの種類 (すべて必須):
1. **単体テスト** — 個々の関数、ユーティリティ、コンポーネント
2. **統合テスト** — API エンドポイント、データベース操作
3. **E2E テスト** — 重要なユーザー フロー

**TDD ワークフロー (必須):**
1. 最初にテストを書き込みます (赤) — テストは失敗するはずです
2. 最小限の実装を書く (緑) — テストは合格するはずです
3. リファクタリング (改善) — カバレッジ 80% 以上を確認する

失敗のトラブルシューティング: テストの分離を確認する → モックを検証する → 実装を修正する (テストが間違っている場合を除き、テストではありません)。

## 開発ワークフロー

1. **計画** — プランナー エージェントを使用し、依存関係とリスクを特定し、フェーズに分割します
2. **TDD** — tdd-guide エージェントを使用し、最初にテストを作成し、実装し、リファクタリングします
3. **レビュー** — コードレビュー担当者を直ちに使用し、重大/高レベルの問題に対処します
4. **知識を適切な場所に取り込む**
   - 個人的なデバッグメモ、環境設定、および一時的なコンテキスト → 自動メモリ
   - チーム/プロジェクトの知識 (アーキテクチャの決定、API の変更、ランブック) → プロジェクトの既存のドキュメント構造
   - 現在のタスクがすでに関連するドキュメントまたはコード コメントを生成している場合は、同じ情報を他の場所に複製しないでください。
   - 明らかなプロジェクト ドキュメントの場所がない場合は、新しいトップレベル ファイルを作成する前に尋ねてください。
5. **コミット** — 従来のコミット形式、包括的な PR 概要

## ワークフロー サーフェス ポリシー

- `skills/` は正規のワークフロー サーフェスだ。
- 新しいワークフローのコントリビューションは、最初に `skills/` に配置される必要がある。
- `commands/` は従来のスラッシュ エントリの互換性サーフェスであり、移行またはクロスハーネス パリティにシムが依然として必要な場合にのみ追加または更新する必要がある。

## Git ワークフロー

**コミット形式:** `<type>: <description>` — タイプ: feat、fix、リファクタリング、docs、test、chore、perf、ci

**PR ワークフロー:** 完全なコミット履歴を分析 → 包括的な概要を草案 → テスト計画を含める → `-u` フラグを使用してプッシュする。

## アーキテクチャ パターン

**API 応答形式:** 成功インジケーター、データ ペイロード、エラー メッセージ、ページネーション メタデータを含む一貫したエンベロープ。

**リポジトリ パターン:** 標準インターフェイス (findAll、findById、作成、更新、削除) の背後でデータ アクセスをカプセル化する。ビジネス ロジックは、ストレージ メカニズムではなく、抽象インターフェイスに依存する。

**スケルトン プロジェクト:** 実戦テストされたテンプレートを検索し、並列エージェント (セキュリティ、拡張性、関連性) で評価し、最適な一致を複製し、実証済みの構造内で反復する。

## パフォーマンス

**コンテキスト管理:** 大規模なリファクタリングや複数ファイル機能を使用する場合は、コンテキスト ウィンドウの最後の 20% を避けてください。機密性の低いタスク (単一の編集、ドキュメント、単純な修正) では、使用率が高くても許容される。

**ビルドのトラブルシューティング:** build-error-resolver エージェントを使用し、エラーを分析し、段階的に修正し、修正するたびに確認する。

## プロジェクトの構造

```
agents/          — 47 specialized subagents
skills/          — 181 workflow skills and domain knowledge
commands/        — 79 slash commands
hooks/           — Trigger-based automations
rules/           — Always-follow guidelines (common + per-language)
scripts/         — Cross-platform Node.js utilities
mcp-configs/     — 14 MCP server configurations
tests/           — Test suite
```

`commands/` は互換性のためにリポジトリに残りますが、長期的な方向性はスキル第一だ。
