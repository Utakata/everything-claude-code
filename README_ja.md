## Quick Start

2分未満で起動・実行できる。

### Pick one path only

ほとんどのClaude Codeユーザーは、正確に1つのインストールパスを使用するべきである。

- **推奨されるデフォルト：** Claude Codeプラグインをインストールし、実際に必要なruleフォルダーのみをコピーする。
- **手動インストーラーを使用する場合：** より細かい制御が必要な場合、プラグインパスを完全に避けたい場合、またはClaude Codeのビルドでセルフホストされたマーケットプレイスエントリの解決に問題がある場合のみ。
- **インストール方法を重ねないこと。** 最も一般的な壊れたセットアップは、最初に`/plugin install`を実行し、その後で`install.sh --profile full`または`npx ecc-install --profile full`を実行することである。

すでに複数のインストールを重ね、重複して見える場合は、[Reset / Uninstall ECC](#reset--uninstall-ecc)に直接進むこと。

### Low-context / no-hooks path

hooksがグローバルすぎると感じる場合や、ECCのrules、agents、コマンド、およびコアワークフローskillsのみが必要な場合は、プラグインをスキップして、最小の（minimal）手動プロファイルを使用する。

```bash
./install.sh --profile minimal --target claude
```

```powershell
.\install.ps1 --profile minimal --target claude
# or
npx ecc-install --profile minimal --target claude
```

このプロファイルは意図的に`hooks-runtime`を除外する。

通常のコアプロファイルが必要だが、hooksをオフにする必要がある場合は、次を使用する。

```bash
./install.sh --profile core --without baseline:hooks --target claude
```

ランタイムでの強制が必要な場合のみ、後でhooksを追加する。

```bash
./install.sh --target claude --modules hooks-runtime
```

### Find the right components first

どのECCプロファイルやコンポーネントをインストールすべきかわからない場合は、任意のプロジェクトからパッケージされたアドバイザーに尋ねること。

```bash
npx ecc consult "security reviews" --target claude
```

一致するコンポーネント、関連プロファイル、プレビュー/インストールコマンドを返す。正確なファイルプランを検査したい場合は、インストールする前にプレビューコマンドを使用する。

本番環境のML/MLOpsワークフローの場合は、インストールをオプトインにして、コンポーネントのスコープを維持する。

```bash
npx ecc consult "mlops training model deployment" --target claude
npx ecc install --profile minimal --target claude --with capability:machine-learning
```

### Step 1: Install the Plugin (Recommended)

> 注: プラグインは便利だが、Claude Codeビルドでセルフホストされたマーケットプレイスエントリの解決に問題がある場合は、以下のOSSインストーラーが依然として最も信頼できるパスである。

```bash
# Add marketplace
/plugin marketplace add https://github.com/affaan-m/everything-claude-code

# Install plugin
/plugin install ecc@ecc
```

### Naming + Migration Note

ECCには現在3つの公開識別子があり、それらは互換性がない。

- GitHub ソースリポジトリ: `affaan-m/everything-claude-code`
- Claude マーケットプレイス/プラグイン識別子: `ecc@ecc`
- npm パッケージ: `ecc-universal`

これは意図的である。Anthropicのマーケットプレイス/プラグインのインストールは、正規のプラグイン識別子によってキー指定されるため、厳格なデスクトップ/APIバリデーターに対応できる短いツール名およびスラッシュコマンド名前空間を維持するために、ECCは`ecc@ecc`を使用している。古い投稿には以前の長いマーケットプレイス識別子が表示されているかもしれないが、それはレガシーなエイリアスとしてのみ扱うこと。別途、npmパッケージは`ecc-universal`に留まっているため、npmでのインストールとマーケットプレイスでのインストールは意図的に異なる名前を使用している。

### Step 2: Install Rules Only If You Need Them

> 警告: **重要:** Claude Codeのプラグインは自動的に`rules`を配布することはできない。
>
> すでに`/plugin install`経由でECCをインストールしている場合、**その後で`./install.sh --profile full`、`.\install.ps1 --profile full`、または`npx ecc-install --profile full`を実行してはならない**。プラグインはすでにECCのskills、コマンド、hooksをロードしている。プラグインのインストール後にフルインストーラーを実行すると、それと同じサーフェスがユーザーディレクトリにコピーされ、重複したskillsやランタイムの動作を引き起こす可能性がある。
>
> プラグインでインストールした場合は、必要な`rules/`ディレクトリのみを`~/.claude/rules/ecc/`以下に手動でコピーすること。まずは`rules/common`と、実際に使用している言語またはフレームワークパックの1つから始める。すべてのコンテキストを明示的にClaudeに組み込みたい場合を除き、すべてのrulesディレクトリをコピーしてはならない。
>
> フルインストーラーは、プラグインのパスを使用せずに完全に手動でECCをインストールする場合にのみ使用すること。
>
> ローカルのClaudeのセットアップがワイプまたはリセットされた場合でも、ECCを再購入する必要はない。まずは`node scripts/ecc.js list-installed`から始め、次に`node scripts/ecc.js doctor`と`node scripts/ecc.js repair`を実行してから何かを再インストールすること。これにより、セットアップを再構築せずにECC管理下のファイルが復元されることが多い。問題がECC Toolsの課金/アカウントアクセスに関するものである場合は、別途課金/アカウントの復旧として対処すること。

```bash
# Clone the repo first
git clone https://github.com/affaan-m/everything-claude-code.git
cd everything-claude-code

# Install dependencies (pick your package manager)
npm install        # or: pnpm install | yarn install | bun install

# Plugin install path: copy only ECC rules into an ECC-owned namespace
mkdir -p ~/.claude/rules/ecc
cp -R rules/common ~/.claude/rules/ecc/
cp -R rules/typescript ~/.claude/rules/ecc/

# Fully manual ECC install path (use this instead of /plugin install)
# ./install.sh --profile full
```

```powershell
# Windows PowerShell

# Plugin install path: copy only ECC rules into an ECC-owned namespace
New-Item -ItemType Directory -Force -Path "$HOME/.claude/rules/ecc" | Out-Null
Copy-Item -Recurse rules/common "$HOME/.claude/rules/ecc/"

# Fully manual ECC install path (use this instead of /plugin install)
# .\install.ps1 --profile full
# npx ecc-install --profile full
```

手動でのインストール手順については、`rules/`フォルダー内のREADMEを参照すること。手動でrulesをコピーする際は、相対参照が機能し続け、ファイル名が衝突しないように、中のファイルではなく言語ディレクトリ全体（例えば`rules/common`や`rules/golang`）をコピーすること。

### Reset / Uninstall ECC

ECCが重複している、押し付けがましい、または壊れていると感じた場合は、自身の上に再インストールを繰り返さないこと。

- **プラグインパスの場合:** Claude Codeからプラグインを削除し、その後、`~/.claude/rules/ecc/`以下に手動でコピーした特定のruleフォルダーを削除する。
- **手動インストーラー / CLIパスの場合:** リポジトリのルートから、まず削除のプレビューを実行する。

```bash
node scripts/uninstall.js --dry-run
```

その後、ECC管理下のファイルを削除する。

```bash
node scripts/uninstall.js
```

ライフサイクルラッパーを使用することもできる。

```bash
node scripts/ecc.js list-installed
node scripts/ecc.js doctor
node scripts/ecc.js repair
node scripts/ecc.js uninstall --dry-run
```

ECCは、インストール状態に記録されたファイルのみを削除する。インストールしていない無関係なファイルは削除しない。

方法を重ねてしまった場合は、次の順序でクリーンアップする。

1. Claude Codeプラグインのインストールを削除する。
2. リポジトリのルートからECCアンインストールコマンドを実行し、インストール状態で管理されているファイルを削除する。
3. 手動でコピーした不要なruleフォルダーを削除する。
4. 1つのパスを使用して1回だけ再インストールする。

## Cross-Platform Support

このプラグインは現在、**Windows、macOS、Linux**を完全にサポートしており、主要なIDE（Cursor、Zed、OpenCode、Antigravity）およびCLIハーネスとも緊密に統合されている。すべてのhooksとスクリプトは、最大限の互換性を得るためにNode.jsで書き直された。

### Package Manager Detection

プラグインは、以下の優先順位で、お好みのパッケージマネージャー（npm、pnpm、yarn、またはbun）を自動的に検出する。

1. **環境変数**: `CLAUDE_PACKAGE_MANAGER`
2. **プロジェクト設定**: `.claude/package-manager.json`
3. **package.json**: `packageManager` フィールド
4. **ロックファイル**: package-lock.json、yarn.lock、pnpm-lock.yaml、またはbun.lockbからの検出
5. **グローバル設定**: `~/.claude/package-manager.json`
6. **フォールバック**: 最初に利用可能なパッケージマネージャー

好みのパッケージマネージャーを設定するには:

```bash
# Via environment variable
export CLAUDE_PACKAGE_MANAGER=pnpm

# Via global config
node scripts/setup-package-manager.js --global pnpm

# Via project config
node scripts/setup-package-manager.js --project bun

# Detect current setting
node scripts/setup-package-manager.js --detect
```

または、Claude Codeで`/setup-pm`コマンドを使用する。

### Hook Runtime Controls

ランタイムフラグを使用して、厳格さを調整したり、特定のhooksを一時的に無効にしたりする。

```bash
# Hook strictness profile (default: standard)
export ECC_HOOK_PROFILE=standard

# Comma-separated hook IDs to disable
export ECC_DISABLED_HOOKS="pre:bash:tmux-reminder,post:edit:typecheck"

# Cap SessionStart additional context (default: 8000 chars)
export ECC_SESSION_START_MAX_CHARS=4000

# Disable SessionStart additional context entirely for low-context/local-model setups
export ECC_SESSION_START_CONTEXT=off

# Keep context/scope/loop warnings but suppress API-rate cost estimates
export ECC_CONTEXT_MONITOR_COST_WARNINGS=off
```

Windows PowerShellの場合:

```powershell
[Environment]::SetEnvironmentVariable('ECC_CONTEXT_MONITOR_COST_WARNINGS', 'off', 'User')
```

Claude Codeの設定をスキャンして、脆弱性、設定ミス、およびインジェクションのリスクを検出する。

```bash
# Quick scan (no install needed)
npx ecc-agentshield scan

# Auto-fix safe issues
npx ecc-agentshield scan --fix

# Deep analysis with three Opus 4.6 agents
npx ecc-agentshield scan --opus --stream

# Generate secure config from scratch
npx ecc-agentshield init
```

**スキャン対象:** CLAUDE.md、settings.json、MCP configs、hooks、agent定義、およびskills。これらは、シークレット検出（14パターン）、権限監査、hookインジェクション分析、MCPサーバーリスクプロファイリング、およびagent configレビューの5つのカテゴリーにわたってスキャンされる。

**`--opus`フラグ**は、レッドチーム/ブルーチーム/監査パイプラインで3つのClaude Opus 4.6 agentsを実行する。攻撃者はエクスプロイトチェーンを見つけ、防御者は保護を評価し、監査者は両方を統合して優先順位付けされたリスク評価を作成する。単なるパターンマッチングではなく、敵対的推論（Adversarial reasoning）を行う。

**出力フォーマット:** Terminal（カラーグレーディングA-F）、JSON（CIパイプライン用）、Markdown、HTML。クリティカルな発見があった場合は終了コード2を返し、ビルドゲートとして機能する。

Claude Codeで`/security-scan`を使用して実行するか、[GitHub Action](https://github.com/affaan-m/agentshield)でCIに追加する。

[GitHub](https://github.com/affaan-m/agentshield) | [npm](https://www.npmjs.com/package/ecc-agentshield)

### Important: Hooks Auto-Loading Behavior

> 警告: **コントリビューター向け:** `.claude-plugin/plugin.json`に`"hooks"`フィールドを追加**しないこと**。これは回帰テストによって強制されている。

Claude Code v2.1以降では、規約によりインストールされたプラグインから`hooks/hooks.json`を**自動的にロード**する。`plugin.json`で明示的に宣言すると、重複検出エラーが発生する。

```

```

**履歴:** これは、このリポジトリで繰り返し修正/元に戻すサイクルを引き起こしてきた（[#29](https://github.com/affaan-m/everything-claude-code/issues/29)、[#52](https://github.com/affaan-m/everything-claude-code/issues/52)、[#103](https://github.com/affaan-m/everything-claude-code/issues/103)）。Claude Codeのバージョン間で動作が変更され、混乱を招いた。現在、これが再導入されるのを防ぐための回帰テストがある。

#### Install hooks

生のままのリポジトリの`hooks/hooks.json`を`~/.claude/settings.json`や`~/.claude/hooks/hooks.json`にコピー**しないこと**。そのファイルはプラグイン/リポジトリ向けであり、ECCインストーラーを通じてインストールするか、プラグインとしてロードすることを意図しているため、生のコピーはサポートされている手動インストールパスではない。

コマンドパスが正しく書き換えられるように、インストーラーを使用してClaude hookランタイムのみをインストールする：

```bash
# macOS / Linux
bash ./install.sh --target claude --modules hooks-runtime
```

```powershell
# Windows PowerShell
pwsh -File .\install.ps1 --target claude --modules hooks-runtime
```

これにより、解決されたhooksが`~/.claude/hooks/hooks.json`に書き込まれ、既存の`~/.claude/settings.json`はそのまま維持される。

`/plugin install`経由でECCをインストールした場合、それらのhooksを`settings.json`にコピーしてはならない。Claude Code v2.1以降はプラグインの`hooks/hooks.json`をすでに自動ロードしており、`settings.json`でそれらを重複させると、二重実行やクロスプラットフォームのhookの競合を引き起こす。

Windowsに関する注意：Claude設定ディレクトリは`~/claude`ではなく`%USERPROFILE%\\.claude`である。

#### Configure MCPs

Claudeプラグインのインストールでは、意図的にECCにバンドルされているMCPサーバー定義を自動有効化しない。これにより、手動でのMCPセットアップを利用可能に保ちつつ、厳格なサードパーティゲートウェイで長すぎるプラグインMCPツール名が使用されるのを防ぐ。

ライブのClaude Codeサーバーの変更には、Claude Codeの`/mcp`コマンドまたはCLI管理のMCPセットアップを使用する。Claude Codeランタイムでの無効化には`/mcp`を使用する。Claude Codeはそれらの選択を`~/.claude.json`に永続化する。

リポジトリローカルでのMCPアクセスの場合は、必要なMCPサーバー定義を`mcp-configs/mcp-servers.json`からプロジェクトスコープの`.mcp.json`にコピーする。

すでにECCバンドルのMCPの独自のコピーを実行している場合は、次のように設定する：

```bash
export ECC_DISABLED_MCPS="github,context7,exa,playwright,sequential-thinking,memory"
```

ECCが管理するインストールおよびCodexの同期フローは、重複して再追加する代わりに、それらのバンドルサーバーをスキップまたは削除する。`ECC_DISABLED_MCPS`はECCのインストール/同期フィルターであり、ライブのClaude Codeのトグルではない。

**重要:** `YOUR_*_HERE`のプレースホルダーを実際のAPIキーに置き換えること。

## Key Concepts

### Agents

サブエージェント（Subagents）は、限定されたスコープで委任されたタスクを処理する。例：

```markdown
---
name: code-reviewer
description: Reviews code for quality, security, and maintainability
tools: ["Read", "Grep", "Glob", "Bash"]
model: opus
---

You are a senior code reviewer...
```

### Skills

Skillsは主要なワークフローサーフェスである。直接呼び出すことも、自動的に提案されることも、agentsによって再利用されることも可能である。ECCは移行中も維持された`commands/`を引き続き提供し、非推奨の短い名前のシムは明示的なオプトイン専用として`legacy-command-shims/`に存在する。新しいワークフロー開発は、まず`skills/`に着地させるべきである。

```markdown
# TDD Workflow

1. Define interfaces first
2. Write failing tests (RED)
3. Implement minimal code (GREEN)
4. Refactor (IMPROVE)
5. Verify 80%+ coverage
```

### Hooks

Hooksはツールイベントで発火する。例 - console.logについて警告する：

```json
{
  "matcher": "tool == \"Edit\" && tool_input.file_path matches \"\\\\.(ts|tsx|js|jsx)$\"",
  "hooks": [{
    "type": "command",
    "command": "#!/bin/bash\ngrep -n 'console\\.log' \"$file_path\" && echo '[Hook] Remove console.log' >&2"
  }]
}
```

### Rules

Rulesは常に従うべきガイドラインであり、`common/`（言語に依存しない）+ 言語固有のディレクトリに編成されている：

```
rules/
  common/          # Universal principles (always install)
  typescript/      # TS/JS specific patterns and tools
  python/          # Python specific patterns and tools
  golang/          # Go specific patterns and tools
  swift/           # Swift specific patterns and tools
  php/             # PHP specific patterns and tools
  arkts/           # HarmonyOS / ArkTS patterns and constraints
```

インストールと構造の詳細については、[`rules/README.md`](rules/README.md)を参照すること。

## Which Agent Should I Use?

どこから始めればよいかわからない場合は、このクイックリファレンスを使用すること。Skillsは正規のワークフローサーフェスである。維持されたスラッシュエントリは、コマンド優先のワークフローで引き続き利用可能である。

| I want to... | Use this surface | Agent used |
|--------------|-----------------|------------|
| 新しい機能の計画を立てる | `/ecc:plan "Add auth"` | planner |
| システムアーキテクチャの設計 | `/ecc:plan` + architect agent | architect |
| テストを先に書いてコードを作成する | `tdd-workflow` skill | tdd-guide |
| 書いたばかりのコードをレビューする | `/code-review` | code-reviewer |
| 失敗したビルドを修正する | `/build-fix` | build-error-resolver |
| E2Eテストを実行する | `e2e-testing` skill | e2e-runner |
| セキュリティの脆弱性を見つける | `/security-scan` | security-reviewer |
| デッドコードを削除する | `/refactor-clean` | refactor-cleaner |
| ドキュメントを更新する | `/update-docs` | doc-updater |
| Goコードをレビューする | `/go-review` | go-reviewer |
| Pythonコードをレビューする | `/python-review` | python-reviewer |
| F#コードをレビューする | *(`fsharp-reviewer`を直接呼び出す)* | fsharp-reviewer |
| TypeScript/JavaScriptコードをレビューする | *(`typescript-reviewer`を直接呼び出す)* | typescript-reviewer |
| HarmonyOSアプリを開発する | *(`harmonyos-app-resolver`を直接呼び出す)* | harmonyos-app-resolver |
| データベースクエリを監査する | *(自動的に委任される)* | database-reviewer |
| 本番MLの変更をレビューする | `mle-workflow` skill + `mle-reviewer` agent | mle-reviewer |

### Common Workflows

以下のスラッシュ形式は、維持されたコマンドサーフェスの一部として残っている場所に示されている。`/tdd`や`/eval`のような非推奨の短い名前のシムは、明示的なオプトイン専用として`legacy-command-shims/`に存在する。

**新しい機能の開始：**
```
/ecc:plan "Add user authentication with OAuth"
                                              → plannerが実装のブループリントを作成する
tdd-workflow skill                            → tdd-guideがwrite-tests-first（テストを先に書くこと）を強制する
/code-review                                  → code-reviewerがあなたの作業をチェックする
```

**バグの修正：**
```
tdd-workflow skill                            → tdd-guide: バグを再現する失敗するテストを書く
                                              → 修正を実装し、テストがパスすることを確認する
/code-review                                  → code-reviewer: リグレッションを捕捉する
```

**本番環境の準備：**
```
/security-scan                                → security-reviewer: OWASP Top 10の監査
e2e-testing skill                             → e2e-runner: 重要なユーザーフローのテスト
/test-coverage                                → 80%以上のカバレッジを検証する
```

<details>
<summary><b>カスタムAPIエンドポイントやモデルゲートウェイ上のClaude CodeでECCを使用できるか？</b></summary>

できる。ECCはAnthropicがホストするトランスポート設定をハードコードしていない。Claude Codeの通常のCLI/プラグインサーフェスを介してローカルで実行されるため、以下で動作する。

- AnthropicがホストするClaude Code
- `ANTHROPIC_BASE_URL`と`ANTHROPIC_AUTH_TOKEN`を使用する公式のClaude Codeゲートウェイのセットアップ
- Claude Codeが期待するAnthropic APIを話す互換性のあるカスタムエンドポイント

最小限の例：

```bash
export ANTHROPIC_BASE_URL=https://your-gateway.example.com
export ANTHROPIC_AUTH_TOKEN=your-token
claude
```

ゲートウェイがモデル名を再マッピングする場合は、ECC内ではなくClaude Code内でそれを設定する。`claude` CLIがすでに機能していれば、ECCのhooks、skills、コマンド、rulesはモデルプロバイダーに依存しない。

公式リファレンス：
- [Claude Code LLM gateway docs](https://docs.anthropic.com/en/docs/claude-code/llm-gateway)
- [Claude Code model configuration docs](https://docs.anthropic.com/en/docs/claude-code/model-config)

</details>

<details>
<summary><b>コンテキストウィンドウが縮小している / Claudeがコンテキスト不足になっている</b></summary>

多すぎるMCPサーバーはコンテキストを食いつぶす。各MCPツールの説明は200kのウィンドウからトークンを消費し、潜在的に~70kまで減らす可能性がある。SessionStartのコンテキストはデフォルトで8000文字に制限されている。ローカルモデルや低コンテキストのセットアップの場合は、`ECC_SESSION_START_MAX_CHARS=4000`で減らすか、`ECC_SESSION_START_CONTEXT=off`で無効にする。

**修正方法:** `/mcp`を使用してClaude Codeから未使用のMCPを無効にする。Claude Codeはそれらのランタイムの選択を`~/.claude.json`に書き込む。`.claude/settings.json`や`.claude/settings.local.json`は、すでにロードされたMCPサーバーに対する信頼できるトグルではない。

有効にするMCPはプロジェクトあたり10未満にし、アクティブなツールは80未満に保つこと。
</details>

### Cursor Loading Notes

ECCは、ルートの`AGENTS.md`を`.cursor/`にインストールしない。Cursorはネストされた`AGENTS.md`ファイルをディレクトリコンテキストとして扱うため、ECCのリポジトリIDをホストプロジェクトにコピーすると、そのプロジェクトが汚染されるためである。

Cursorのネイティブな読み込み動作は、Cursorのビルドによって異なる場合がある。ECCはagentsを`.cursor/agents/ecc-*.md`としてインストールする。Cursorのビルドがプロジェクトのagentsを公開していない場合でも、これらのファイルは隠されたグローバルなプロンプトコンテキストとしてではなく、明示的な参照定義として機能する。

### Hook Architecture (DRY Adapter Pattern)

Cursorは**Claude Codeよりも多くのhookイベント**を持っている（20対8）。`.cursor/hooks/adapter.js`モジュールは、Cursorの標準入力JSONをClaude Codeのフォーマットに変換し、既存の`scripts/hooks/*.js`を重複なしで再利用できるようにする。

```
Cursor stdin JSON → adapter.js → transforms → scripts/hooks/*.js
                                              (shared with Claude Code)
```

主要なhooks：
- **beforeShellExecution** — tmux外の開発サーバーをブロック（exit 2）、git pushのレビュー
- **afterFileEdit** — 自動フォーマット + TypeScriptチェック + console.log警告
- **beforeSubmitPrompt** — プロンプト内のシークレット（sk-、ghp_、AKIAパターン）を検出
- **beforeTabFileRead** — Tabキーによる.env、.key、.pemファイルの読み取りをブロック（exit 2）
- **beforeMCPExecution / afterMCPExecution** — MCP監査ログ

### Rules Format

Cursorのrulesは、`description`、`globs`、`alwaysApply`を含むYAMLフロントマターを使用する：

```yaml
---
description: "TypeScript coding style extending common rules"
globs: ["**/*.ts", "**/*.tsx", "**/*.js", "**/*.jsx"]
alwaysApply: false
---
```

### Quick Start (GitHub Copilot)

ファイルはすでに配置されている。このプロジェクトを含む任意のリポジトリを開くと、GitHub Copilot Chatは自動的に`.github/copilot-instructions.md`を認識する。
コミットされた`.vscode/settings.json`は`chat.promptFiles`を有効にするため、VS Codeは`.github/prompts/`から再利用可能なプロンプトを読み込むことができる。

Copilot Chatでワークフロープロンプトを使用するには：
1. VS CodeでCopilot Chatパネルを開く。
2. **ペーパークリップ / 添付**アイコンをクリックして**Prompt...**を選択するか、`/`と入力してプロンプトを選択する。
3. プロンプトを選択する（例：`plan`、`tdd`、`code-review`）。

### How It Works

VS CodeのGitHub Copilotは、2種類のファイルを自動的に読み込む：

- **`.github/copilot-instructions.md`** — リポジトリレベルの命令。すべてのCopilot Chatリクエストに常に注入される。ECCのコアなコーディング標準、セキュリティチェックリスト、テスト要件、およびgitワークフローが含まれている。
- **`.github/prompts/*.prompt.md`** — ユーザーがオンデマンドで呼び出す再利用可能なプロンプトファイル。各プロンプトは、特定のECCワークフロー（計画 → TDD → レビュー → デプロイ）を通じてCopilotを導く。

**`.vscode/settings.json`**はタスクごとの命令オーバーレイを追加し、コードを生成しているのか、テストを書いているのか、選択範囲をレビューしているのか、コミットメッセージを下書きしているのかに応じて、Copilotが適切なコンテキストを受け取るようにする。

### Feature Coverage

| ECC Feature | Copilot equivalent |
|-------------|-------------------|
| Coding standards | `copilot-instructions.md`を介して常にオン |
| Security checklist | 常にオン + `security-review` prompt |
| Testing / TDD | 常にオン + `tdd` prompt |
| Implementation planning | `plan` prompt |
| Code review | `code-review` prompt |
| Build error resolution | `build-fix` prompt |
| Refactoring | `refactor` prompt |
| Commit message format | `settings.json`のタスクごとの命令 |
| Hooks / automation | サポートされていない（Copilotにはhookシステムがない） |
| Agents / delegation | サポートされていない（CopilotにはサブエージェントAPIがない） |

### Limitations

GitHub CopilotにはhookシステムやサブエージェントAPIがないため、ECCのhook自動化（自動フォーマット、TypeScriptチェック、セッションの永続化、開発サーバーガード）とエージェントへの委任は利用できない。命令とプロンプトレイヤーは、標準、セキュリティ、TDD、ワークフローといった完全なECCのコーディング哲学をすべてのCopilot Chatセッションにもたらす。

## Token Optimization

トークンの消費を管理しないと、Claude Codeの使用は高額になる可能性がある。これらの設定は、品質を犠牲にすることなくコストを大幅に削減する。

### Recommended Settings

`~/.claude/settings.json`に追加する：

```json
{
  "model": "sonnet",
  "env": {
    "MAX_THINKING_TOKENS": "10000",
    "CLAUDE_AUTOCOMPACT_PCT_OVERRIDE": "50"
  }
}
```

| Setting | Default | Recommended | Impact |
|---------|---------|-------------|--------|
| `model` | opus | **sonnet** | コストを約60%削減; コーディングタスクの80%以上を処理する |
| `MAX_THINKING_TOKENS` | 31,999 | **10,000** | リクエストごとの隠れた思考コストを約70%削減する |
| `CLAUDE_AUTOCOMPACT_PCT_OVERRIDE` | 95 | **50** | より早く圧縮する — 長いセッションでの品質が向上する |
| `ECC_CONTEXT_MONITOR_COST_WARNINGS` | on | **サブスクリプションユーザーの場合はoff** | コンテキスト/スコープ/ループの警告を維持しつつ、エージェント向けAPIレート見積もりの警告を抑制する |

深いアーキテクチャの推論が必要な場合のみOpusに切り替える：
```
/model opus
```

### Daily Workflow Commands

| Command | When to Use |
|---------|-------------|
| `/model sonnet` | ほとんどのタスクのデフォルト |
| `/model opus` | 複雑なアーキテクチャ、デバッグ、深い推論 |
| `/clear` | 無関係なタスクの間（無料、即時リセット） |
| `/compact` | 論理的なタスクの区切り（調査完了、マイルストーン完了） |
| `/cost` | セッション中のトークン消費を監視する |

Claudeのサブスクリプションを使用しており、コンテキストモニターのAPIレート見積もりが役に立たない場合は、`ECC_CONTEXT_MONITOR_COST_WARNINGS=off`に設定する。これはエージェント向けのコスト警告を抑制するだけであり、コンテキストの枯渇、スコープ、またはループの警告を無効にするものではない。

### Strategic Compaction

このプラグインに含まれる`strategic-compact` skillは、95%コンテキストでの自動圧縮に依存するのではなく、論理的な区切りで`/compact`を提案する。完全な決定ガイドについては、`skills/strategic-compact/SKILL.md`を参照のこと。

**いつ圧縮するか:**
- 調査/探索の後、実装の前
- マイルストーンを完了した後、次のマイルストーンを開始する前
- デバッグの後、機能の作業を再開する前
- アプローチが失敗した後、新しいアプローチを試す前

**いつ圧縮しないか:**
- 実装の途中（変数名、ファイルパス、部分的な状態が失われる）

### Context Window Management

**重要:** すべてのMCPを一度に有効にしないこと。各MCPツールの説明は200kのウィンドウからトークンを消費し、潜在的に~70kまで減らす可能性がある。

- プロジェクトあたり有効にするMCPは10未満に保つ
- アクティブなツールは80未満に保つ
- `/mcp`を使用してClaude Codeの未使用のMCPサーバーを無効にする。これらのランタイムの選択は`~/.claude.json`に永続化される。
- `ECC_DISABLED_MCPS`は、インストール/同期フロー中にECCが生成したMCP設定をフィルタリングするためだけに使用する。

### Agent Teams Cost Warning

Agent Teamsは複数のコンテキストウィンドウを生成する。各チームメイトは独立してトークンを消費する。並列処理が明確な価値を提供するタスク（マルチモジュールの作業、並行レビュー）にのみ使用すること。単純な連続的なタスクの場合、サブエージェントの方がトークン効率が良い。

### Token Optimization

1日の制限に達している場合は、推奨される設定とワークフローのヒントについて**[Token Optimization Guide](docs/token-optimization.md)**を参照のこと。

クイックウィン（即効性のある対策）：

```json
// ~/.claude/settings.json
{
  "model": "sonnet",
  "env": {
    "MAX_THINKING_TOKENS": "10000",
    "CLAUDE_AUTOCOMPACT_PCT_OVERRIDE": "50",
    "CLAUDE_CODE_SUBAGENT_MODEL": "haiku"
  }
}
```

無関係なタスクの間では`/clear`を使用し、論理的な区切りでは`/compact`を使用し、消費を監視するために`/cost`を使用する。
